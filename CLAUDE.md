## What this is

**smartmet-library-grid-files** — a C++ library providing a unified interface for reading GRIB1, GRIB2, NetCDF, and QueryData grid files. It acts as a driver/adapter layer so that upper libraries and services can access any grid format through the same API without knowing the underlying file structure.

Part of the SmartMet Server ecosystem (Finnish Meteorological Institute).

## Build commands

```bash
make                # Build libsmartmet-grid-files.so
make debug          # Build with debug flags
make test           # Run tests (in test/ subdirectory)
make clean          # Clean build artifacts
make rpm            # Build RPM package
make install        # Install library and headers (PREFIX=/usr)
make create_def     # Regenerate auto-generated GRIB definition classes
make delete_def     # Clean auto-generated definitions
```

No test directory exists in this repository — tests are currently absent.

## Dependencies

Build requires installed: smartmet-macgyver, smartmet-newbase, smartmet-spine, smartmet-trax, GDAL, GEOS, Boost (iostreams, thread), libpng, libjpeg, openjpeg2, libaec, libcurl, libwebp.

`REQUIRES = geos gdal webp filesystem` in the Makefile drives pkg-config resolution.

## Architecture

### Class hierarchy (adapter pattern)

```
GRID::GridFile          — Entry point; memory-maps a file, detects format, creates Messages
  └─ GRID::Message      — Abstract base class (~250 methods): metadata, grid geometry,
                           data retrieval, interpolation, coordinate conversions, contouring
       ├─ GRIB1::Message
       ├─ GRIB2::Message
       ├─ NetCDF::Message
       └─ QueryData::Message
```

`GRID::Message` (src/grid/Message.h) is the central abstraction — all format-specific classes override its virtual methods. `GRID::GridFile` auto-detects the file format and creates the appropriate message type.

### Source modules (src/)

| Directory | Purpose |
|-----------|---------|
| `grid/` | Format-independent interface: `GridFile`, `Message`, `Typedefs`, `MessageProcessing`, value caches |
| `common/` | Utilities: memory mapping (`MemoryMapper` with userfaultfd), binary I/O (`MemoryReader`), bit-level access, interpolation functions, coordinate conversions, threading primitives, image/graph functions, configuration parsing |
| `grib1/` | GRIB edition 1: section classes (Indicator, Product, Grid, Bitmap, Data), packing algorithms |
| `grib2/` | GRIB edition 2: section classes (Indicator, Identification, Local, Grid, Product, Representation, Bitmap, Data), more packing algorithms |
| `grib1/definition/`, `grib2/definition/` | **Auto-generated** C++ classes for grid/product templates (see below) |
| `grib1/implementation/`, `grib2/implementation/` | Manual `*Impl` extension classes that add logic on top of auto-generated definitions |
| `identification/` | Parameter/geometry/level mapping engine. `GridDef` is the central registry mapping between GRIB, NetCDF, Newbase, and FMI identifiers |
| `netcdf/` | NetCDF adapter — lightweight wrapper creating `Message` objects from NetCDF variables |
| `querydata/` | QueryData adapter — wraps FMI's native `NFmiQueryData` format |
| `map/` | Topography support: land/sea masks, elevation shading |

### Auto-generated code

Files in `src/grib1/definition/` and `src/grib2/definition/` are **generated by PHP scripts** in `php/` from eccodes GRIB API template definitions (`/usr/share/eccodes/definitions/`). Do not edit these files directly.

- `php/createClassDefinitions_grib1.php` → generates from `grib1/grid_definition_*.def`
- `php/createClassDefinitions_grib2.php` → generates from `grib2/grid_definition_*.def` and product templates

To regenerate: `make create_def` (requires eccodes definitions installed).

Manual extensions live in the `implementation/` directories as `*Impl.h`/`*Impl.cpp` files — these are safe to edit.

### FMI identifier system

All format-specific parameters ultimately map to FMI identifiers, which serve as the lingua franca across formats. The mapping chain:

- GRIB1/2 fields → GRIB ID → FMI parameter ID
- Newbase ID → FMI parameter ID
- NetCDF variable name → FMI parameter ID

Mappings are defined in CSV files under `cfg/` and loaded at runtime via `cfg/grid-files.conf`. The `%(DIR)` placeholder in the config resolves to the cfg directory path. Extension files in `cfg/ext/` allow site-specific overrides.

### Key design details

- **Memory mapping with userfaultfd**: `MemoryMapper` supports lazy page-fault-driven loading from local and remote (HTTP/S3) sources, with prefetching and authentication
- **Coordinate caching**: Expensive projection transformations are cached for repeated access
- **Format detection**: `GridFile` reads magic bytes to identify format (GRIB="GRIB" with edition byte, NetCDF="CDF"/"\x89HDF", QueryData=FMI header)
- **Geometry groups**: Support for time-varying grid resolutions via geometry group definitions

## .clang-format

The `.clang-format` in this repo has `DisableFormat: true` — clang-format is intentionally disabled. The Makefile `format` target references a different path (`$(SUBNAME)/*.h`) that doesn't match the actual source layout, so formatting is effectively not enforced.
